## 2026-05-20 修复：摸鱼笔记高亮 Bug + TodoPanel 显示问题

**Bug 1：笔记正文不显示**
- 原因：`buildHighlightHtml` 从 `el.innerText` 读取源文本，但 contenteditable div 刚创建时 innerText 为空
- 修复：改用 `noteContent.value` 作为源文本

**Bug 2：只显示最后一条高亮**
- 原因：无位置的高亮用文本匹配时，逐一从剩余文本中移除匹配内容，截断了后续搜索
- 修复：合并定位（`lastIndexOf`）+ 重叠去重，一次排序渲染

**Bug 3：syncContentEditable 调用未定义函数**
- 原因：`refreshContenteditable` 函数不存在
- 修复：补全函数定义，委托给 `buildHighlightHtml`

**Bug 4：刷新后高亮位置丢失**
- 原因：highlight start 位置只在内存中，API 未持久化
- 修复：Todo 模型新增 `highlight_start` 列，存入选文本在笔记中的起始偏移
- 前端 `fetchLinkedTodoIds` 从 API 获取位置
- 已为旧数据执行 backfill（7条记录）

**文件变更**：
- `backend/models/todo.py` — 新增 `highlight_start` 列 + to_dict
- `backend/routes/todo.py` — create/update 支持 `highlight_start`
- `src/stores/scratch-note.js` — createTodo 传 `highlight_start` + fetch 用 API 位置
- `src/components/center/ScratchNote.vue` — 修复3个函数

## 2026-05-20 修复2：删除待办后高亮未消失

**原因**：从右侧 TodoPanel 删除词条待办后，`todo-changed` → `scratch-note-refresh-highlights` 事件触发了 `buildHighlightHtml()`，但 store 的 `highlightedTexts` 还是旧数据（未同步 API）。

**修复**：
- `onRefreshHighlights` 先调 `ensureLinkedTodoIds()` 刷新 store，再调 `buildHighlightHtml()`
- `openNote` 也先调 `ensureLinkedTodoIds()`，确保打开笔记时高亮数据是最新的

## 2026-05-20 修复3：词条待办点击跳转到笔记正文

**原因**：右侧 TodoPanel 中带 note_id 的待办只有 🔗 按钮可跳转，点击文字/📝图标无响应。

**修复**：在 sticky-note div 上添加 `@click="onNoteClick(todo)"`，有 note_id 的跳转到笔记正文页，有 notebook_id 的跳转到笔记本目录。checkbox 和 按钮已有 `@click.stop`，互不影响。添加 `.is-note` 类指针样式。

## 2026-05-20 修复4：第二次点击跳转到目录页（终版）

**根因**：事件时序问题。Home.vue 用 dispatchEvent 通知 ScratchNote 跳转，但组件挂载（含事件监听器注册）和 init() 都是异步的，事件可能在监听器注册前就丢了。多次尝试（setTimeout、nextTick、提前注册监听器）均不可靠。

**终版修复**：放弃事件机制，改用 Pinia store 直接传参。
- Store 新增 `pendingOpenNoteId` 字段
- Home.vue `onJumpToNote` 直接设 `store.pendingOpenNoteId = noteId`
- ScratchNote `onMounted` 在 init() 前捕获此值，init 完成后调用 `openNote`
- 同时添加 `watch(() => store.pendingOpenNoteId)` 处理已挂载时再次跳转的场景
